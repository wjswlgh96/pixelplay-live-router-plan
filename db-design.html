기존 채팅 애플리케이션 도메인은 유지하고, 라이브 MVP만 새 도메인으로 확장합니다.

테이블현재 역할라이브 MVP 영향결정
public.user프로필, OAuth 계정 기준, 닉네임과 아바타 표시 기준입니다.크리에이터, 시청자, 후원자, 투표 참여자의 공통 기준 테이블입니다.재사용합니다.

public.chat_room_member일반 채팅방 멤버십과 강퇴 상태입니다.라이브는 비로그인 시청과 공개 읽기 플로우가 있어 일반 채팅방 멤버십과 분리합니다.유지합니다.
public.message일반 채팅 메시지입니다.라이브 채팅은 방송 회차, 클린봇 결과, 후원 이벤트와 연결되므로 별도 live_message를 사용합니다.유지합니다.

설계 원칙

planning MVP와 실제 project RPC convention을 같이 맞춘 기준입니다.

  • 별도 creator_channel 테이블은 만들지 않습니다. MVP는 1유저 1라이브 채널이므로 /live/[creatorId]의 creatorId는 public.user.id를 그대로 사용합니다.
  • 설정과 방송 회차는 분리합니다. creator_studio_setting은 크리에이터당 1개 row를 계속 update하고, live_broadcast는 방송을 켤 때마다 새 row를 생성합니다.
  • 현재 방송 여부는 상태 enum이 아니라 ended_at is null로 판단합니다. 공개/비공개 방송 기능은 MVP에서 제거했으므로 live_visibility도 만들지 않습니다.
  • 일반 채팅과 라이브 채팅은 분리합니다. 일반 메시지는 message, 라이브 채팅과 클린봇 결과는 live_message에 저장합니다.
  • 팔로우와 채팅 규칙 동의는 같은 관계 테이블에 둡니다. viewer_creator_relation은 팔로우 여부, 팔로우 시각, 채팅 규칙 동의 버전을 같이 관리합니다.
  • 후원은 지갑 원장과 방송 후원 이벤트를 나눕니다. 금액 정합성은 wallet_transaction이 담당하고, 방송 화면과 크리에이터 수익 내역은 donation이 담당합니다.
  • OBS Alert 템플릿은 DB에 저장하지 않습니다. MVP는 고정 템플릿이므로 DB에는 Alert 사용 여부, 볼륨, TTS 시도 여부, URL 버전만 저장합니다.
  • 실제 정산 인증은 MVP DB 범위에서 제외합니다. 디자인 프리뷰와 테스트 계정 표시용 settlement_demo만 저장하고, 실명조회와 계좌 인증은 후속 확장으로 둡니다.

Toss 결제 API 설계

Toss Payments는 MVP 결제사로 확정하고, 클라이언트 결제창과 서버 원장 반영을 분리합니다.

라우터입력서버 책임DB 반영
/api/payments/toss/prepare충전 금액로그인, 금액 옵션, orderId 형식, idempotencyKey를 생성하고 Toss 결제창에 넘길 payload를 반환합니다.필요 시 wallet_transaction에 pending 충전 row를 먼저 남깁니다.
/api/payments/toss/confirmpaymentKey, orderId, amount요청 금액과 주문 주체를 검증한 뒤 Toss /v1/payments/confirm을 호출합니다. 승인 상태가 DONE일 때만 지갑 반영 RPC를 호출합니다.confirm_wallet_charge로 payment_key, order_id, idempotency_key, balance_after를 저장합니다.
/api/payments/toss/webhookToss 이벤트 payload서명 검증 후 결제 실패, 취소, 승인 지연을 idempotency 기준으로 보정합니다.이미 성공 처리된 충전은 중복 반영하지 않고, 실패·취소는 wallet_transaction_status만 보정합니다.
